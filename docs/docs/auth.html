Auth

One policy. Pages, REST, and MCP.

Auth in Apex is a single, server-side, fail-closed policy applied to every surface — page loaders, REST /api/*, and the MCP /mcp endpoint — so an AI calling your tools can never do more than the logged-in user. Neither Next nor Nuxt governs the AI surface this way.

Setup #

Scaffold the resolver and a login/logout pair, then set a session secret:

apex make auth   # → server/auth.ts + server/api/login.ts + logout.ts

apex.config.ts — a 32+ char secret (override via APEX_SESSION_PASSWORD)

import { defineConfig } from '@apex-stack/core'

export default defineConfig({
  runtimeConfig: { sessionPassword: process.env.APEX_SESSION_PASSWORD },
})

Identify the user #

defineAuth in server/auth.ts answers "who is this request?" once per request. Its result is injected as user into every loader, route handler, and MCP tool call — null means anonymous. Return whatever shape you like.

server/auth.ts

import { useRuntimeConfig } from '@apex-stack/core'
import { sessionAuth } from '@apex-stack/core/server'

// Reads the sealed session cookie written by /api/login. Or write your own
// defineAuth({ resolve }) to read a JWT / header / adapter session.
export default sessionAuth({ password: String(useRuntimeConfig().sessionPassword) })

Gate a route #

Add auth: true to require a user, and an optional can for fine-grained checks on the user and the validated input: 

import { defineApexRoute } from '@apex-stack/core'

export default defineApexRoute({
  method: 'DELETE',
  auth: true,                                  // anonymous → 401
  can: ({ user }) => user.roles.includes('admin'),  // not allowed → 403
  handler: ({ user }) => removeThing(user),
})

REST returns 401 (no user) or 403 (not permitted). Over MCP, a route the caller can't reach is omitted from tools/list per-user and refused on tools/call — enforced at the handler layer, never middleware-only.

Resource access & scope #

For defineResource/defineModel, gate each operation with access and filter rows per-caller with scope. scope is applied to every op — a WHERE on list/get/update/delete and a stamp on create — so a caller only ever sees or touches their own rows.

models/note.ts

import { defineModel } from '@apex-stack/data'

export default defineModel('note', {
  fields: { ownerId: 'string', body: 'text' },
  access: { list: 'authed', get: 'authed', create: 'authed', update: 'authed', delete: 'authed' },
  scope: ({ user }) => ({ ownerId: user.id }),   // row-level: only your rows
})
Fail-closed by default

Declaring access or scope gates the whole resource — any op you don't list defaults to authed. You can't half-secure a resource and leave delete open. create stamps ownership from scope (a client can't spoof ownerId), and update can't reassign a scoped column. Guessing another user's id returns null — no enumeration oracle.

Sessions #

The built-in session is a sealed (encrypted + signed) cookie — HttpOnly, Secure, SameSite=Lax. Route handlers receive event; pass it to the session helpers. Anonymous requests get no cookie.

server/api/login.ts

import { defineApexRoute, useRuntimeConfig } from '@apex-stack/core'
import { login, setStatus } from '@apex-stack/core/server'
import { z } from 'zod'

export default defineApexRoute({
  method: 'POST',
  input: { email: z.string(), password: z.string() },
  handler: async ({ input, event }) => {
    const user = await verify(input)          // your credential check → user | null
    if (!user) { setStatus(event, 401); return { error: 'Invalid credentials' } }
    await login(event, { user }, { password: String(useRuntimeConfig().sessionPassword) })
    return { ok: true }
  },
})

logout(event, { password }) clears the cookie. Prefer an adapter (below) for real credential storage, OAuth, or 2FA.

Roles & permissions #

When a flat user.roles array isn't enough, createAccessControl gives you a persisted, Spatie-style RBAC: roles hold permissions, users hold roles, and a user can also be granted a permission directly. It runs against a DB handle you pass, and its effective-permission check is the union of role-granted and direct grants.

server side — @apex-stack/core/server

import { createAccessControl } from '@apex-stack/core/server'

const ac = createAccessControl({ handle })   // handle = your DB adapter
await handle.exec(ac.migrationSql())          // roles, permissions, role_permissions, user_roles

await ac.grantPermissionToRole('editor', 'posts.edit')
await ac.assignRole('u1', 'editor')
await ac.giveDirectPermission('u1', 'billing.view')

await ac.permissionsOf('u1')               // → ['billing.view', 'posts.edit']
await ac.hasPermission('u1', 'posts.edit')   // → true
await ac.hasRole('u1', 'editor')            // → true

await ac.revokeRole('u1', 'editor')          // grants & assignments are revocable + idempotent

permissionGate(ac, 'posts.edit') adapts a permission into a route/resource can — it reads the user's id, checks the store, and is fail-closed for anonymous callers: 

import { defineApexRoute } from '@apex-stack/core'
import { permissionGate } from '@apex-stack/core/server'

export default defineApexRoute({
  method: 'POST',
  auth: true,
  can: permissionGate(ac, 'posts.edit'),   // no user / no permission → refused
  handler: ({ user }) => publishPost(user),
})
Injection-safe by construction

The table prefix is validated as a SQL identifier (createAccessControl throws on a bad prefix), and every value is a bound ? parameter — nothing is string-concatenated into SQL. A user with no roles or grants simply has no permissions (no empty-IN () crash).

API tokens #

For non-cookie clients — a CLI, a mobile app, a CI job — createTokenStore issues opaque, Sanctum-style bearer tokens with per-token abilities. issue returns the plaintext once; the store persists only a SHA-256 hash, never the secret. 

import { createTokenStore } from '@apex-stack/core/server'

const tokens = createTokenStore({ handle })
await handle.exec(tokens.migrationSql())

const { id, token } = await tokens.issue('u1', 'cli', ['posts.read', 'posts.write'])
// `token` is shown to the user once — only its hash is stored.

await tokens.verify(token)              // → { id, userId: 'u1', abilities: [...] } | null
await tokens.can(token, 'posts.read')    // → true  (the '*' ability matches anything)
await tokens.list('u1')                 // summaries only — never the hash
await tokens.revoke(id)                 // or tokens.revokeAll('u1')

verify returns null — never throws — for an unknown id, a wrong secret, a malformed string, or an expired token (pass { expiresInSeconds } to issue); a successful verify stamps last_used_at.

Auth flow tokens #

Password-reset and email-verification links need single-use, expiring tokens. createFlowTokens issues one bound to a kind and a subject (usually an email); consume succeeds exactly once and then can never be replayed. 

import { createFlowTokens } from '@apex-stack/core/server'

const flows = createFlowTokens({ handle, ttlSeconds: 3600 })
await handle.exec(flows.migrationSql())

const token = await flows.issue('password_reset', 'user@example.com')
// …email the link containing `token`…

await flows.consume('password_reset', 'user@example.com', token)   // → true (first time)
await flows.consume('password_reset', 'user@example.com', token)   // → false (single-use)

A wrong secret, wrong kind, wrong subject, expired, or garbage token all return false without throwing — and a wrong-kind attempt never burns the real token. email_verify and password_reset are independent flows for the same subject, and only a hash is stored.

The AI surface #

Because every typed route is also an MCP tool, auth has to reach the AI too — and it does, from the same policy. Per request, the tool list is filtered to what the caller may reach, and every tools/call re-checks authorization with the real arguments (defense-in-depth). A scoped resource returns only the caller's rows over MCP exactly as over REST.

Why this matters

Next and Nuxt secure pages and API routes, but neither has an AI-callable surface to govern. Apex's moat — every route is an MCP tool — would be a liability without this: here the same auth/can/access/scope that guards REST guards the tools your AI can call. See AI-native APIs.

Hardening #

Because Apex owns the server, sensible defaults ship in @apex-stack/core/server:

ConcernWhat you get
CSRFOrigin/Referer check on cookie-authenticated mutations (bearer/tokenless clients are exempt). Automatic in /api.
SessionsSealed cookies — HttpOnly, Secure, SameSite=Lax; no cookie for anonymous callers.
Rate limitingcreateRateLimiter({ limit, windowMs }) — pure, clock-injectable; key by IP with rateLimitKey(event). Pass a shared store (any KvStore) for one global counter across instances. 🟡 Experimental
IdempotencySend Idempotency-Key on a POST/PUT/PATCH/DELETE — automatic in /api: the handler runs once and retries replay the cached response (x-idempotent-replay: true); a concurrent duplicate gets 409; 5xx isn't cached, so retries re-execute. Keys are scoped per route + user. 🟡 Experimental
HeaderssecurityHeaders() / applySecurityHeaders(event) — conservative defaults, add your own CSP.
SSR islandLoaders decide what serializes; never bake secrets into the hydration state.

Both are backed by one pluggable primitive — KvStore, a tiny async key/value store with TTL. The built-in createMemoryStore() is correct for a single process; on serverless / multi-instance, implement KvStore over Redis (or any shared store) and pass it in: 

import { createProdApp, createRateLimiter, type KvStore } from '@apex-stack/core/server'

const redisStore: KvStore = { /* get / set / incr(ttl on create) / delete */ }

// One global rate-limit counter across every instance:
const limiter = createRateLimiter({ limit: 100, windowMs: 60_000, store: redisStore })

// Idempotency replay shared across instances:
const app = await createProdApp({ dir: 'dist', idempotencyStore: redisStore })

Adapters #

OAuth providers, JWT issuance, and 2FA are their own products — Apex doesn't reinvent them. Bring an adapter (Lucia, Better-Auth, Auth.js) and surface its user from defineAuth: read its session in resolve, or map its payload with sessionAuth's toUser. Everything above — route gating, resource access, scope, the MCP surface — works unchanged on top of whatever identity you return.
