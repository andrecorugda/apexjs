One definition, two surfaces #

defineApexRoute takes a Zod input shape, a handler, and a description. That single definition drives REST validation and the MCP tool's inputSchema — no duplication, no drift.

server/api/add.ts

import { defineApexRoute } from '@apex-stack/core'
import { z } from 'zod'

export default defineApexRoute({
  method: 'POST',
  description: 'Add two numbers and return their sum', // → MCP tool description
  input: { a: z.number(), b: z.number() },            // → REST validation + MCP inputSchema
  mcp: true,                                          // opt in to MCP exposure
  handler: ({ input }) => ({ sum: input.a + input.b }),
})

You now have both surfaces from that one file:

defineApexRoute
server/api/add.ts
↓ generates ↓
REST endpoint
POST /api/add {a,b} → { "sum": 5 }
MCP tool
tools/call add {a,b} → { "sum": 5 }

Invalid input returns 400 with the Zod issues. Only mcp: true routes are exposed as tools — everything else stays REST-only.

defineApexRoute options #

OptionTypePurpose
method'GET' | 'POST' | 'PUT' | 'PATCH' | 'DELETE'HTTP method. Defaults to GET.
descriptionstringHuman + AI-readable. Becomes the MCP tool description.
inputZod raw shapeAn object of Zod validators. Drives REST validation and the MCP inputSchema.
mcpbooleanOpt in to MCP exposure. Defaults to false.
mcpNamestringOverride the tool name (must match ^[a-zA-Z0-9_-]{1,64}$). Defaults to a slug from the file path.
handler(ctx) => OutputThe implementation. ctx.input is validated; ctx.url is the raw request URL.
Keep handlers thin

A handler is an adapter — business logic belongs in a service so it's reusable beyond the route. The route's job is the schema-carrying contract that makes REST + MCP automatic.

For a GET route, input is read from the query string; for others, from the JSON body. Use z.coerce.* when a query value should become a number/boolean:

server/api/greet.ts

export default defineApexRoute({
  method: 'GET',
  description: 'Greet a person by name',
  input: { name: z.string() },
  mcp: true,
  handler: ({ input }) => ({ message: `Hello, ${input.name}!` }),
})

The /mcp endpoint #

Apex serves all mcp: true routes at a single /mcp endpoint using the official MCP SDK's Web-standard Streamable HTTP transport, in stateless JSON mode — a fresh server per request, no session bookkeeping. It's available in apex dev and, in production, from apex build --server + apex start.

  • tools/list returns every exposed route as a tool, with its description and inputSchema.
  • tools/call <name> runs the same handler as the REST endpoint, over the same data.

The apex mcp inspector #

A built-in inspector connects to a running dev server's /mcp, lists tools with their signatures, and calls one — the local test loop for AI-callable routes, right in your terminal (it reuses the bundled MCP SDK, so there's nothing to install). 

# list every tool and its input signature
apex mcp

# call one with JSON arguments
apex mcp --call add --args '{"a":2,"b":3}'

# point at a non-default endpoint
apex mcp --url http://localhost:4000/mcp

Full flags on the CLI reference. Smoke-testing tools with the inspector before shipping is the recommended rhythm.

Point Claude at your API #

Because /mcp speaks standard MCP over Streamable HTTP, any MCP client — Claude included — can connect to it and call your tools directly. Run your app, expose /mcp, and an assistant can read and write through the very same handlers your UI uses. No adapter, no second implementation, no drift between "the API" and "what the AI can do".

Build with an agent (CLI as MCP) #

Everything above makes your running app AI-callable. Apex flips the other way too: the CLI itself is an MCP server, so an agent can build your app — scaffold pages, add models, wire up auth, run the build — by calling tools instead of guessing at file contents and shelling out. Run it over stdio: 

# stdio MCP server exposing the Apex CLI as tools — for an AI agent host
apex mcp-server

# operate on a specific project (agent hosts launch from their own cwd)
apex mcp-server --root /abs/path/to/your-app

It writes nothing but MCP protocol to stdout — no banner, no logs — so it's safe as a stdio transport. Tools that fail return isError: true, so the agent can tell a failed call from a successful one.

The server exposes the generator, build, and introspection commands as eight tools:

ToolDoesKey inputs
apex_makeGenerate a file — page, component, api, service, store, layout, middleware, test, model, migration, auth.kind, name, fields
apex_extendAdd an optional feature: data, auth, or i18n. Idempotent.feature
apex_addCopy in themeable UI components. force: true overwrites existing files.components, force
apex_buildBuild the app, optionally with a deploy preset (vercel | netlify | docker) or a Node server target.preset, server
apex_listDiscover what's available: features, components, or make-kinds.what
apex_project_infoRead before you write — page routes; API routes with method, mounted path, and mcp/auth flags; models with their table, fields, and behaviors; components; and installed features. Enough shape to act without opening files.root
apex_checkType-check the app (tsc --noEmit; uses the native compiler when installed). The fast type gate — run it after generating code, before tests. Fails with the type errors when there are any.root
apex_testRun the app's tests once with Vitest, optionally filtered.pattern
Why tools beat a plain shell

The tools carry typed inputs and structured results, so the agent scaffolds correct, conventional code instead of hand-writing boilerplate it can get subtly wrong. apex_project_info lets it read the app's real shape first; apex_check then apex_test let it verify its own change. Read → generate → check → test, all as tool calls — and with the native TypeScript compiler the check is fast enough to run after every edit.

The AGENTS.md context pack #

Every new app is scaffolded with an AGENTS.md — a concise brief that teaches an agent how to work in an Apex project before it touches a file. It's the same idea as this page, distilled for a coding agent:

  • Golden rules — prefer generators over hand-writing, one place for everything, let the framework own security, verify with tests, lean on strict TypeScript.
  • The generator table — every apex make/extend/add command and what it produces.
  • Project structure — what each folder is for, so the agent never invents one.
  • The .alpine format and the core APIs — routes, models, auth, testing.

Agent hosts that auto-discover AGENTS.md (Cursor, and others) pick it up with no configuration. Already have an app from an older scaffold? apex upgrade writes the latest AGENTS.md in, non-destructively.

Connect Claude Desktop or Cursor #

Point any MCP-capable agent host at apex mcp-server and it gains the eight tools above. With the CLI installed globally (npm i -g @apex-stack/core):

claude_desktop_config.json / Cursor MCP settings

{
  "mcpServers": {
    "apex": {
      "command": "apex",
      "args": ["mcp-server", "--root", "/abs/path/to/your-app"]
    }
  }
}

Not installed globally? Launch it through the project's local binary with npx, using the package explicitly so it resolves from anywhere: 

{
  "mcpServers": {
    "apex": {
      "command": "npx",
      "args": ["--package", "@apex-stack/core", "apex", "mcp-server", "--root", "/abs/path/to/your-app"]
    }
  }
}

Give --root an absolute path — the host launches the server from its own working directory, not your project's. Restart the host, and the agent can scaffold, build, and test your Apex app conversationally.

Resources are AI-native too #

The same principle scales to data. defineResource turns one table into list/get/create/update/delete over REST and as MCP tools (<name>_list, <name>_create, …) on one database — so an AI creating a row and a browser listing rows see the same data.

Design notes #

  • The schema is the enabler. The Zod shape is used for REST validation and the MCP inputSchema — one source of truth.
  • Opt-in & safe. MCP exposure is per-route (mcp: true). Free-form handlers without a schema aren't auto-exposed, by design.
  • Stateless. The /mcp transport runs in stateless JSON mode — a fresh server per request.
  • Access-filtered. A route's auth/can gate applies to its MCP tool too — tools the caller can't reach are omitted from tools/list and refused on call, so the AI surface is scoped exactly like REST.
On the roadmap

outputSchema → MCP structured content, and extracting the MCP server into its own package to keep core lean.
