Build & deploy

Three build modes, from fully static to full server.

apex build spans a spectrum — a zero-JS static site, a prerendered-and-hydrated static bundle, or a deployable Node server for dynamic routes, API, MCP, and a database. Pick the most static mode your app can use.

Choosing a mode #

ModeCommandShipsUse when
Static (SSG)apex build --islandsHTML, zero framework JS (islands hydrate where present)Content sites; static routes only.
Prerendered + hydratedapex buildHTML + per-page client bundleInteractive apps that can run from a static host.
Server targetapex build --server + apex startClient bundles + SSR modules + a Node serverDynamic routes, API/MCP, or a database in production.

1 · Static site (SSG) — apex build --islands #

Prerenders every static route to dist/**/index.html with zero framework JS — islands hydrate lazily where present. Deploy dist/ to any static host. 

apex build --islands
# → dist/index.html, dist/about/index.html, …  (static-first)

2 · Prerendered + hydrated — apex build #

Prerenders each page to HTML and builds a per-page client bundle (hashed, with a shared runtime chunk). Pages are server-rendered for first paint, then hydrate — fully interactive from a static dist/. 

apex build
# → dist/  (prerendered HTML + hashed client bundles)
Dynamic routes need the server target

apex build and --islands prerender only static routes and report any dynamic [param] routes they skipped. To serve those in production, use --server below.

3 · Server target — apex build --server + apex start #

Builds client bundles and SSR modules plus a run manifest (apex-manifest.json), producing a deployable Node server: 

apex build --server     # → dist/ (client assets + dist/server/*.mjs + apex-manifest.json)
apex migrate            # apply DB migrations (deploy step)
apex start              # run the production server (no Vite)

The production server (apex start, verified end-to-end) serves:

  • Dynamic routes — pages/blog/[slug].alpine rendered per request.
  • API + MCP — server/api/* as REST endpoints and MCP tools at /mcp.
  • Static assets — hashed client bundles with long-cache headers.
  • Any database — SQLite/Turso/Supabase/Neon via @apex-stack/data.

One-command deploys #

apex build --preset <target> generates the host config for a server-target deploy. Set DATABASE_URL and APEX_SESSION_PASSWORD in the host's environment, then deploy.

Vercel — serverless

apex build --preset vercel   # writes api/index.mjs + vercel.json
vercel deploy

Netlify — serverless

apex build --preset netlify  # writes netlify/functions/server.mjs + netlify.toml
netlify deploy

Docker — Railway / Render / Fly.io / any container or VM

apex build --preset docker   # writes a Dockerfile (+ .dockerignore)
# push to your host — apex start listens on the injected $PORT
How it works

The serverless presets wrap the built app in one function via createProdNodeHandler (Vercel) / createProdWebHandler (Netlify) from @apex-stack/core/server; the Docker preset builds and runs apex start. All serve SSR + /api + /mcp from the same server target — one build, any host.

Databases in production #

Set DATABASE_URL in your host's env and createDb connects — one driver switch, no code changes:

await createDb({ driver: 'postgres', url: process.env.DATABASE_URL }) // Supabase / Neon / RDS
await createDb({ driver: 'libsql',   url: process.env.TURSO_URL })    // Turso (hosted libSQL)
ProviderDriverNotes
SupabasepostgresApp: the transaction pooler URL (:6543) — Apex auto-sets prepare:false + SSL. Migrations: the session/direct URL (:5432).
NeonpostgresServerless Postgres; use the pooled connection string.
Railway / AWS RDS / any PostgrespostgresStandard connection string.
TursolibsqlHosted libSQL — libsql://… URL + auth token.
SQLite file / PGlitelibsql / pgliteSingle Node host / local / embedded.
PlanetScale / MySQL—Not supported yet — Apex's data layer is SQLite + Postgres.

Run apex migrate as a deploy step (with the matching --driver/--url). See Data → Drivers.

Production reliability # 🟡 Experimental

The server target ships the reliability basics a real deploy needs — no configuration required:

ConcernWhat you get
Health checksGET /health and /healthz return 200 {"status":"ok","uptime":…} — liveness only (no DB ping), served before auth/middleware. Point your platform's probe at it.
Graceful shutdownOn SIGTERM/SIGINT, apex start stops accepting connections, drains in-flight requests (10s window, then force-close), and closes database pools — @apex-stack/data handles register their own close(). A second Ctrl+C force-exits.
Safe 500sClients never see error messages or stack traces in production — a thrown API handler or page loader returns a generic Internal server error; the full detail goes to your onError hook (or the server log). Dev keeps the rich errors.
Request loggingOne JSON line per request: {"time","method","path","status","ms"} (health probes are skipped). Silence with apex start --quiet or APEX_LOG=off.

Wire your own observability with server/hooks.ts — discovered like server/auth.ts, baked into the build: 

// server/hooks.ts — e.g. Sentry + your log shipper
import { defineHooks } from '@apex-stack/core/server'

export default defineHooks({
  onError(error, ctx) {
    // ctx.kind: 'api' | 'page' | 'mcp' | 'http' (+ path, method)
    Sentry.captureException(error, { tags: ctx })
  },
  onRequest(entry) {
    // replaces the default JSON line — entry: {time, method, path, status, ms}
    myLogger.info(entry)
  },
  async onShutdown() {
    // runs after in-flight requests drain (flush queues, telemetry, …)
    await Sentry.close(2000)
  },
})

Embedders (serverless / custom servers) get the same pieces programmatically: createProdApp({ hooks, requestLog }), gracefulShutdown(server), and onShutdown(fn) from @apex-stack/core/server.
