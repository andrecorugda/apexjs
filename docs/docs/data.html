Data

One table → REST and MCP CRUD.

@apex-stack/data turns one defineModel into a full resource: list, get, create, update, delete — over REST and as MCP tools, on the same database. The same model is an Eloquent-style active-record API for your own code. Your data is AI-callable by construction.

Install #

Add the data package (and Drizzle) alongside core:

npm i @apex-stack/data drizzle-orm

Postgres/PGlite drivers load on demand — install only the one you use (below).

Define once #

One defineModel is the single source of truth: from a field spec it derives the Zod validation, the dialect-aware Drizzle table, the CREATE TABLE migration SQL, a full REST + MCP resource, and a typed active-record query API. Composable behaviors passed to use (timestamps(), owned(), softDeletes(), auditable(), observable()) fold in lifecycle logic and row-level security.

models/Message.ts

import { defineModel, timestamps } from '@apex-stack/data'

export const Message = defineModel('messages', {
  fields: {
    author: { type: 'string', notNull: true },
    body: { type: 'string', notNull: true },
  },
  use: [timestamps()], // created_at / updated_at, kept out of the insert shape
})

Open one shared database handle in db/index.ts. Use lazyDb — it opens without a top-level await (so the module also bundles into the on-device apex build --mobile output) and connects, migrates, and seeds on first use. A file-backed libSQL/SQLite database locally (data + migration history persist); Postgres when DATABASE_URL is set:

db/index.ts

import { applyMigrations, lazyDb } from '@apex-stack/data'
import { Message } from '../models/Message.js'

const url = process.env.DATABASE_URL
const onDevice = (globalThis as { __APEX_DEVICE__?: boolean }).__APEX_DEVICE__
export const handle = lazyDb(
  () => (url ? { driver: 'postgres', url } : { driver: 'libsql', url: 'file:./data.db' }),
  { init: async (h) => {
    // Server/dev: run the tracked db/migrations/*.sql files. On-device (no fs): create from the model.
    onDevice ? await h.exec(Message.migrationSql(h.dialect)) : await applyMigrations(h, 'db/migrations')
  } },
)

Mount the model as a live resource from a server/api/*.ts file — this one line is the whole backend:

server/api/messages.ts

import { handle } from '../../db/index.js'
import { Message } from '../../models/Message.js'

export default Message.resource(handle)

apex make model <Name> field:type … scaffolds the model, its resource, a reversible migration, and db/index.ts (if absent) for you.

The surfaces you get #

From that single defineModel definition:

ActionRESTMCP tool
ListGET /api/messagesmessages_list
GetGET /api/messages/:idmessages_get
CreatePOST /api/messagesmessages_create
UpdatePATCH /api/messages/:idmessages_update
DeleteDELETE /api/messages/:idmessages_delete

REST and MCP run the same handlers over the same database. An AI that creates a message via messages_create and a browser that lists via GET /api/messages see the same rows. The model's field spec validates creates; updates accept the same fields, all optional, keyed by id.

List is paginated, filterable, and sortable out of the box — for both REST and the _list MCP tool. Add ?page & ?perPage to get a { data, total, page, perPage, lastPage } envelope (perPage capped at 100; without them list returns a plain array), ?sort=-createdAt,author to sort (leading - = descending), and ?<col>=value to filter by any of the model's columns.

Why this matters

No other full-stack framework gives you REST + a live MCP CRUD surface from one table definition, on one database, by default. See AI-native APIs for how the MCP transport is carried for you.

Query with active record #

The model object is the query API — no hand-written SQL. Reads run through Drizzle (typed columns hydrate boolean/json automatically) and return hydrated instances; .all() returns a Collection. Pass the shared handle as the terminal argument: 

import { raw } from '@apex-stack/data'
import { handle } from '../../db/index.js'
import { Player } from '../../models/Player.js'

await Player.first(handle)            // first row, or null
await Player.find(handle, id)         // by primary key (findOrFail throws if missing)
await Player.all(handle)              // a Collection of every row
await Player.count(handle, { team: 'A' })

// Chainable builder — ops: eq/ne/gt/gte/lt/lte/like/in/notIn/isNull, plus .orWhere()
await Player.where({ team: 'A', plays: { gt: 5, lte: 100 } })
  .orderBy('plays', 'desc').limit(10).offset(0)
  .all(handle)                       // terminals: .all/.first/.count/.exists/.pluck/.sum/avg/min/max/.delete

Writes go through the same pipeline as the REST/MCP resource, so create/update/updateOrCreate/delete fire the model's hooks (timestamps(), observable(), auditable()), apply row-level scope, respect softDeletes(), and validate the payload — so Model.* and /api/* never diverge. Pass { user } as the last argument to apply tenant scope. 

await Player.create(handle, { handle: 'ada', plays: 0 })                  // → the inserted instance
await Player.updateOrCreate(handle, { handle: 'ada' }, { plays: 10 })
await Player.update(handle, id, { plays: raw('plays + 1') })          // raw() = a trusted SQL expr
await Player.delete(handle, { team: 'A' })                            // soft-delete aware; returns the count

// Bulk primitives for imports/ETL (fast; bypass per-row hooks, like Eloquent's bulk ops)
await Player.insertMany(handle, [{ handle: 'a' }, { handle: 'b' }])
await Player.updateMany(handle, { team: { in: ['A', 'B'] } }, { active: false })
await Player.upsert(handle, ['handle'], { handle: 'ada', plays: 99 }, { keep: { plays: 'max' } })

Instances carry their own methods, and a Collection extends Array with Eloquent-style helpers: 

const p = await Player.find(handle, id)
p.plays = 5
p.isDirty()          // true; p.changes() → { plays: 5 }
await p.save()        // persists only dirty attributes through the write pipeline
await p.refresh()     // reloads attributes from the row; p.delete() removes it
p.toJSON()           // omits any `hidden` columns (also stripped from every REST/MCP response)

const players = await Player.all(handle)
players.pluck('handle')         // string[]
players.groupBy('team')          // { A: Collection, B: Collection }
players.sum('plays')

A mistyped column is validated against the model's fields and throws — never an injection vector; values are always bound. Need a join, GROUP BY, or a window function the builder can't express? Player.tableFor(handle) returns the model's Drizzle table for the full typed builder: handle.db.select(…).from(Player.tableFor(handle))….

Relations & eager loading #

Declare relations with thunk refs, then eager-load them with .with(…) — each relation is batch-loaded in ONE query keyed by the parent keys (N parents ⇒ 1 extra query, not N). Loaded relations attach as instance properties and serialize with toJSON. 

import { defineModel, belongsTo, hasMany } from '@apex-stack/data'

export const Post = defineModel('posts', {
  fields: { title: { type: 'string', notNull: true }, authorId: 'int' },
  relations: {
    author: belongsTo(() => User, 'authorId'),
    comments: hasMany(() => Comment, 'postId'),
  },
})

const posts = await Post.with('author', 'comments').orderBy('id', 'asc').all(handle)
posts[0].author    // a User instance
posts[0].comments  // a Collection of Comment instances

hasOne, hasMany, and belongsTo are supported; an unknown relation name throws.

Transactions #

handle.transaction(fn) runs fn atomically — auto-commits when it resolves, auto-rolls-back if it throws. The callback receives a handle bound to the transaction; pass it to your writes so the whole unit is one atomic step: 

await handle.transaction(async (tx) => {
  await Account.create(tx, { name: 'a', balance: 100 })
  await Account.update(tx, id, { balance: raw('balance - 100') })
  // a throw here rolls back BOTH writes
})

Scopes, casts & locking #

Reusable query fragments, attribute casts, optimistic locking, and typed errors round out the Eloquent parity — all declared on the model: 

const Post = defineModel('posts', {
  fields: { title: 'string', status: 'string', at: 'timestamp', prefs: 'text', version: { type: 'int', default: 0 } },
  // Named, chainable scopes → Model.scope('published').all(handle)
  scopes: {
    published: (q) => q.where({ status: 'published' }),
    top: (q, n) => q.orderBy('score', 'desc').limit(n),
  },
  // Value casts on reads/writes: stored as strings, seen as Date / object
  casts: { at: 'date', prefs: 'json' },
  // instance.save() guards the loaded version + bumps it (409 on a stale write)
  optimisticLock: 'version',
})

await Post.scope('published').all(handle)
await Post.scope('top', 5).all(handle)

Missing rows throw ModelNotFoundException (HTTP 404) via findOrFail/firstOrFail; a driver failure is rethrown as a QueryException carrying the model, op, and original cause; a stale save() under optimisticLock throws StaleModelException (HTTP 409) instead of silently clobbering. All extend ApexDataError.

Consume it on the client #

The model drives the client too. apex make composable Post reads models/Post.ts and generates a typed data-hook at composables/usePosts.ts — items / current / loading / error plus fetch / find / create / update / remove, all bound to the model's REST resource, with the field types lifted from defineModel: 

// composables/usePosts.ts (generated)
import { createResourceClient, type ResourceClientState } from '@apex-stack/core/client'

export interface Post { id: number; title: string; published: boolean }
export type NewPost = Omit<Post, 'id'>

export function usePosts(): ResourceClientState<Post, NewPost> {
  return createResourceClient<Post, NewPost>('posts')
}

Spread it into an x-data and drive the UI straight from the model:

<script client>
  import { usePosts } from '../composables/usePosts'
</script>
<template x-data="{ ...usePosts(), init() { this.fetch() } }">
  <template x-for="p in items" :key="p.id"><li x-text="p.title"></li></template>
</template>

Prefer to skip the file? Call createResourceClient('posts') inline — the generated wrapper just adds the types. One defineModel now spans schema, migration, REST, MCP, and the browser.

Databases & drivers #

The handle is driver-agnostic — every driver uses Drizzle's async API, so a model written once runs unchanged across all of them. lazyDb's factory just returns the config for the database you want (the same config shapes createDb accepts): 

{ driver: 'libsql', url: 'file:./data.db' }              // local SQLite file (libSQL) — the default
{ driver: 'libsql', url: ':memory:' }                    // ephemeral (on-device / quick demos)
{ driver: 'libsql', url: process.env.TURSO_URL }        // Turso (edge)
{ driver: 'postgres', url: process.env.DATABASE_URL }   // Supabase / Neon
{ driver: 'pglite' }                                    // embedded Postgres (tests)

Tests reach for the synchronous createDb directly (e.g. await createDb({ driver: 'pglite' })); your app shares the one lazyDb handle from db/index.ts.

DriverDialectGood forInstall
sqlite / libsqlSQLiteLocal dev, single Node host@libsql/client
libsql (Turso)SQLiteEdge — fetch-based, works on Workers@libsql/client
postgresPostgresSupabase / Neon / any Postgrespostgres
pglitePostgresEmbedded Postgres for tests (in-memory)@electric-sql/pglite

SQLite and Turso share a dialect; Supabase, Neon, and PGlite are all Postgres. Edge targets can't run native SQLite — pair them with Turso or Supabase/Neon, whose drivers are fetch-based.

Migrations #

Write plain SQL files in db/migrations/. applyMigrations runs each file once in filename order, tracked in an _apex_migrations table — applied on boot in dev, or explicitly as a deploy step with apex migrate (idempotent).

db/migrations/0001_init.sql

CREATE TABLE IF NOT EXISTS messages (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  author TEXT NOT NULL,
  body TEXT NOT NULL,
  created_at TEXT,
  updated_at TEXT
);
apex migrate                                   # default: data.db (sqlite)
apex migrate --driver postgres --url $DATABASE_URL

Beyond CRUD #

Need something a table can't express — a search, an aggregate, a domain action? Write a plain typed route with defineApexRoute right next to your resources. It's REST + MCP too, so your custom endpoints join the same AI-callable surface.
